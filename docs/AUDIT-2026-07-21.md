# Hearth 4.9.0 architecture and security audit

Date: 2026-07-21

## Scope

Reviewed the Electron main/preload boundary, local JSON storage, renderer chat/call/file flows,
PeerJS identity and connection lifecycle, the optional signal/presence server, Supabase client/RLS,
build dependencies and Playwright smoke tests.

## Resolved in 4.9.0

- Renderer sandbox, navigation/popup denial and trusted-window media permission checks.
- Active-user validation on profile, settings, friends, groups, chat, pins and avatar IPC.
- Atomic local JSON writes and constant-time local password verification.
- Random authenticated cloud peer IDs; unknown peers, identity mismatches and untrusted call metadata are rejected.
- No automatic friend creation from incoming P2P connections.
- Bounded chat/profile/presence inputs and allow-listed remote GIF/media URLs.
- Capability-based file access, 512 MB maximum, strict offsets/chunk sizes and SHA-256 verification.
- Call-end microphone/AudioContext cleanup and correct 0% audio settings.
- Group-call membership synchronization and validation.
- Presence status bug, duplicate username takeover, stale sockets, payload/rate limits and optional shared token.
- Supabase friendship acceptance and group membership RLS flaws; idempotent policy definitions.
- Current Electron/build/test toolchain and automated syntax/signal/UI checks.

## Remaining risks and roadmap

1. **Identity in local-only mode:** deterministic PeerJS IDs do not provide cryptographic identity when the
   authenticated cloud profile is disabled. For internet use, cloud mode is recommended. The next identity
   phase should sign ephemeral peer IDs with a long-term device key and verify signatures on both data and
   media connections.
2. **NAT traversal:** the default ICE list has STUN but no owned TURN service. Calls will fail on symmetric
   NAT or restrictive networks. Deploy coturn with short-lived authenticated credentials and add an ICE
   diagnostics page.
3. **Presence authentication:** the shared signal token is appropriate for a private friend deployment but
   is not per-user authorization and can be extracted from a distributed client. Replace it with short-lived
   Supabase-signed/JWT tickets before operating a public service.
4. **Supabase deployment:** `cloud/schema.sql` is source configuration; its new functions and RLS policies
   must still be applied to the live Supabase project. Presence currently uses a common realtime channel;
   private friend-scoped presence should replace it.
5. **Mesh scaling:** group voice is O(n²). Keep groups small (currently bounded to 12 records); for larger
   rooms use an SFU such as LiveKit/mediasoup rather than extending the mesh.
6. **Storage:** chat history and media previews are plaintext JSON and base64. Move attachments/thumbnails to
   a managed per-user store, add quotas/retention, and optionally encrypt local history with an OS-protected key.
7. **Maintainability:** `public/app.js` is still a large coordinator. Split identity/connections, messages,
   transfers, calls/media and UI state into testable modules before adding major features.
8. **Release trust:** Windows packages are not Authenticode-signed. Add code signing and a CI release workflow
   with protected secrets, reproducible checks and generated checksums.
9. **Coverage:** UI and signal security smoke tests exist, but two-device P2P voice/screen/file integration,
   interrupted-transfer behavior and live Supabase RLS tests should be added next.
